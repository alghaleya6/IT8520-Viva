Data & Features Q&A

Click any question to expand the answer.

Feature selection & cleaning
HighYou started with 56 features but trained on 50. What exactly happened to the other 6?
▼
Removed 7 identifier columns: src_ip, dst_ip, src_port, dst_port, flow_start_time, pcap_source, protocol.

Removed 3 redundant columns (correlation above 0.99): fwd_header_len with fwd_packet_count at 0.9999, bwd_header_len with bwd_packet_count at 0.9998, avg_packet_size with pkt_len_mean at 1.0000.

Added 5 new behavioral features: bytes_per_packet, fwd_bwd_ratio, bwd_bytes_ratio, packet_size_spread, tcp_window_range.
56 minus 10 plus 5 = 51 total columns. Minus 1 label column = 50 input features
HighWhy remove src_ip and dst_ip? Aren't they useful for identifying attackers?
▼
If we keep IPs, the model learns "192.168.85.10 = attack" instead of learning what attack behavior looks like. In any other network with different IPs, the model fails completely. We want the model to learn behavioral patterns — high packet rate, uniform sizes, no backward traffic — that generalize to any network.
Generalization vs memorization. Always remove identifiers from ML training data.
HighWhat is the most important feature and why does it make sense?
▼
fwd_std_packet_len with importance 0.3183 explaining 31.83% of all classification decisions alone.

Why it makes sense: Flood attacks send thousands of identical packets. Standard deviation of packet size is near zero. Normal traffic has natural variation across HTTP, SSH, FTP and DNS. One feature perfectly separates floods from normal traffic.

Top 5 together explain 86.86% of all XGBoost decisions.
MediumWhy clip outliers at 99th percentile instead of removing those rows?
▼
Flood attacks by definition have extreme values in bytes_per_sec and packets_per_sec. Removing those rows means removing real attack data. Clipping caps extreme values without losing any rows. We keep 100% of 101,825 flows while preventing distortion of model training.
Example: if 99% of flows have bytes_per_sec below 500,000, a flood showing 50,000,000 gets capped at 500,000. The row stays but the number is tamed.
MediumWhy class weights instead of SMOTE?
▼
Two reasons we rejected SMOTE. First: data leakage risk — with our temporal split, synthetic samples could straddle the train/test boundary contaminating evaluation. Second: we had enough real samples — 5,669 real Attack rows are sufficient without fake data.

Class weights achieve the same goal: Attack=4.5008, DOS=0.5543. Forces model to pay 4.5x more attention to the rare Attack class.
MediumWhy is StandardScaler needed?
▼
Features have wildly different ranges: bytes_per_sec can be millions while syn_ratio is 0 to 1. Without scaling, high-range features dominate calculations unfairly. StandardScaler transforms every column to mean=0 and std=1.

Especially critical for KNN which uses Euclidean distance. Unscaled features make bytes_per_sec appear millions of times more important than syn_ratio.
Always fit only on training data. Apply the same fitted scaler to test. Fitting on test data = data leakage.

Models Q&A

Questions on training, evaluation, and the why XGBoost defense.

Model selection & evaluation
HighAll 5 models got ~99% accuracy. Why choose XGBoost over Decision Tree when DT had HIGHER F1?
▼
DT scored 0.9987 vs XGBoost 0.9986 — a difference of only 0.0001. We chose XGBoost for five deployment reasons:
1. Most stable: CV std of 0.000103 vs DT 0.000136
2. Best Attack class: Precision 0.9941 and Recall 0.9889
3. Fast inference: nearly instantaneous per flow
4. Small size: only 1.31 MB deployable on a monitoring node
5. predict_proba() enables our confidence-based 3-tier alerts
Decision Trees are high-variance. Small data changes produce very different trees. XGBoost's 200-tree ensemble is inherently more robust in production.
HighPrecision vs Recall — which is more important in cybersecurity?
▼
Precision: of all flows labeled attack, how many were real? (TP divided by TP+FP)
Recall: of all real attacks, how many did we catch? (TP divided by TP+FN)

Recall is more critical in cybersecurity. A missed attack (False Negative) means a real intrusion goes undetected and could bring down smart city services. A false alarm is annoying but manageable.

XGBoost Attack Recall = 0.9889, meaning it catches 98.89% of all real attacks.
HighWhat is cross-validation and what did the std deviation tell you?
▼
Cross-validation checks that test results are not a lucky split. We split training data into 5 equal folds. Each round: train on 4, test on 1. Every sample is tested exactly once. Stratified means each fold maintains the same class distribution as the full dataset.

Low standard deviation means consistent performance regardless of which data the model sees.
XGBoost: std=0.000103 (smallest = most stable). KNN: std=0.000776 (largest = most sensitive to data distribution changes).
MediumWhat is n_estimators=200 in XGBoost?
▼
XGBoost builds 200 sequential trees where each new tree corrects the errors of the previous ones. Combined with max_depth=6 and learning_rate=0.1 this prevents overfitting. The train/test gap was only 0.0011 confirming no overfitting.
Analogy: 200 people each correcting the previous person's essay, but only allowed to make small 10% changes and limited to 6 levels of depth.
MediumWhy did Logistic Regression perform worst? Is that overfitting?
▼
Logistic Regression draws straight decision boundaries. Network traffic classification has curved, complex boundaries. You cannot separate DOS from RECON with a straight line in feature space.

This is underfitting, not overfitting. Proof: training accuracy was 0.9896 and test accuracy was 0.9882, both low with only a 0.0014 gap. If it were overfitting, training would be high but test would be low.
LowWhat does random_state=42 do?
▼
Seeds the random number generator for reproducibility. Without it, shuffling and tree-building produce different results every run making comparisons unreliable. The value 42 is completely arbitrary. What matters is using the same value across all models so any differences in performance are due to the algorithm, not randomness.

IDS Code Q&A

Deep questions on IDS_V2.py — Alghaleya's contribution area.

Architecture & implementation
HighWalk me through step by step from a packet arriving to an alert in the GUI.
▼
1. Scapy sniff() captures a packet on the Host-Only network interface
2. on_packet() (Block 5) extracts: src_ip, dst_ip, protocol, ports, TCP flags, TTL, packet length
3. Packet is assigned to a flow using a 5-tuple key plus a 10-second time window
4. Flow statistics are updated (packet count, bytes, flags, timestamps)
5. When a flow completes via FIN/RST or worker_flush() times it out after 10 seconds:
6. build_features() extracts 50 features in the exact same order as during training
7. StandardScaler normalizes the features using the saved scaler.pkl
8. XGBoost predict_proba() returns probabilities: [attack%, dos%, normal%, recon%]
9. Heuristic overrides check for port scanning or flood patterns
10. Alert tier is assigned: 85%+ = CONFIRMED, 60-85% = SUSPICIOUS, below 60% = IGNORED
11. Event is pushed to the event_pipe queue
12. Tkinter GUI drains the queue every 400ms and updates the live threat log
HighWhat is a race condition and how did you prevent it?
▼
A race condition happens when two threads try to read or write the same data simultaneously causing corrupted or unpredictable results. Our IDS runs 4 threads at once: Scapy packet capture via on_packet, worker_flush, worker_cleanup, and the Tkinter GUI. All 4 access the shared flow_table and sess_table.

Solution: threading.Lock() mutexes. Before accessing shared data we use "with flow_mutex:" which blocks all other threads until the current operation finishes.
HighWhat is worker_flush() and why is it necessary?
▼
Not all flows end with a FIN or RST packet. UDP has no connection termination handshake. SYN flood attacks never complete the TCP handshake. ICMP has no connection concept at all. Without worker_flush, these flows would sit in flow_table forever and never get classified, meaning the IDS would miss them entirely.

worker_flush() runs every 2 seconds and forces classification on any flow where current time minus last packet time is greater than or equal to FLOW_TTL which is 10 seconds.
This is why UDP flood Scenario 2 was detected at 100% confidence. Every UDP flow got classified after the 10-second window expired.
HighWhat are heuristic overrides and why did you need them on top of XGBoost?
▼
Rule-based checks that run alongside XGBoost for patterns so obvious they do not need machine learning:

Port sweep RECON: if src_ip_dst_ports is 15 or more unique destination ports, force "recon" at 90%+ confidence. Catches Nmap scans perfectly.
SYN flood DOS: if syn_ratio above 0.90 AND ack_count equals 0, force "dos". Pure SYN with no ACKs = textbook SYN flood.
UDP flood DOS: if protocol is UDP AND bwd_ratio below 0.05, force "dos". UDP with almost no backward traffic = flood.

This explains why Scenarios 2 and 4 were detected at 100% confidence.
MediumWhy 5 separate .pkl files instead of one?
▼
Each file serves a completely separate purpose at IDS startup:
xgboost_model.pkl (1.31 MB) — the classifier itself
scaler.pkl — must match training stats exactly or features are on wrong scale
label_encoder.pkl — converts 0 to attack for GUI display
feature_columns.pkl — 50 names IN ORDER. Wrong order = completely wrong predictions
thresholds.pkl — stored separately so security teams can tune sensitivity without retraining the model

Attack simulation & lab design
HighYour accuracy is 99.87%. Isn't that too perfect to be trustworthy?
▼
We expected these results. Three pieces of evidence confirm they are valid:
1. Controlled lab environment — SYN floods look completely different from normal SSH. Feature space is naturally well-separated with clean, labeled data
2. Overfitting check passed — XGBoost gap=0.0011. Memorizing would show a much larger gap
3. Cross-validation confirmed it — 5-fold mean=0.9993 with std=0.000103. Cannot be a lucky split

We acknowledge in Section 9.3 that real-world performance with encrypted traffic, IoT devices, and novel attacks would be lower. These are appropriate results for a lab proof-of-concept.
HighIf you deployed this in a real smart city, what would fail?
▼
1. Encrypted HTTPS/TLS — cannot inspect payload, metadata-only features lose discriminative power
2. IoT device traffic — different patterns than what we trained on
3. Novel attacks — zero-days the model has never seen
4. Static model — no automatic retraining as attack patterns evolve
5. Single monitoring node — real smart cities have multiple subnets
6. High throughput — millions of flows per hour, scalability was not tested
HighExplain SYN flood and how your IDS detects it.
▼
TCP 3-way handshake: Client sends SYN, Server replies SYN-ACK, Client sends ACK to complete. In SYN flood: attacker sends thousands of SYN packets but NEVER sends the final ACK. Server allocates memory waiting for ACKs that never come, exhausting its connection table and denying service to legitimate users.

Detection: syn_ratio above 0.90 AND ack_count equals 0 triggers the SYN flood heuristic override which forces DOS classification at 90%+ confidence.

Command used: sudo hping3 -S --flood -V -p 80 192.168.85.20
HighIDS vs IPS — which did you build?
▼
IDS (Intrusion Detection System): passive, monitors traffic, detects threats, generates alerts. Does NOT block traffic.
IPS (Intrusion Prevention System): active, blocks malicious traffic, drops packets, modifies firewall rules in real time.

We built an IDS. Our system classifies flows and displays alerts but does not modify iptables or drop packets. In Section 9.2 we proposed how it could become an IPS: CONFIRMED alerts could auto-trigger iptables commands via Python subprocess calls.

Why temporal split — 10 second answer

"Random split puts packets from the same attack session in both train and test causing data leakage. Temporal per-file split sorts each PCAP chronologically and always tests on newer traffic the model has never seen, simulating real IDS deployment."
